# App - Architecture

## Domain Resources

The App Domain contains the following main resources:

- A React + Vite SPA deployed via AWS Amplify Hosting
- An AppSync GraphQL API auto-generated from the Amplify Data Schema
- A DynamoDB `Job` table (Amplify-managed, PAY_PER_REQUEST, TTL on `expiresAt`) that stores per-job state shared between the app and the scraper domain
- The Amplify Gen 2 backend definition that provisions all of the above

The app is a pure consumer of every other domain: it submits to the scraper's
Validator Lambda Function URL and otherwise reads state from the `Job` row in
DynamoDB via AppSync.

## Real-time updates

The frontend subscribes to a single `Job` row by id via Amplify Data's
`client.models.Job.observeQuery({ id })`. AppSync handles the subscription
plumbing; DynamoDB Streams are not used.

**Backend writers call AppSync mutations directly.** The Validator Lambda and
the Batch worker both call AppSync over HTTPS, authenticating with the
Amplify-issued AppSync API key (`x-api-key` header), and invoke `createJob` /
`updateJob` mutations. The frontend's subscription fires whenever any writer
mutates the row.

## Authorization model

The `Job` schema declares a single authorization mode:

- `allow.publicApiKey()` — used by the browser SPA, the Validator Lambda, and
  the Batch worker. The API key is generated by Amplify, shipped in the
  Amplify-generated `amplify_outputs.json` for the SPA, and written to SSM
  Parameter Store (`/reviewlensai/appsync/apiKey`) for the scraper domain to
  consume.

This is a deliberate PoC trade-off: the API key is exposed in the FE bundle,
which means anyone with the key can read or mutate the `Job` table. The PoC
holds no PII and no sensitive data; this is acceptable. A real app would
introduce Cognito + per-mutation Schema-level `allow.resource(...)` or
`allow.authenticated('userPools')` to lock down writes to trusted
principals.

The original spec called for `allow.authenticated('iam')` on backend writers
plus SigV4 signing. That was simplified during  execution because the
Amplify Gen 2 IAM auth pathway requires either a Cognito Identity Pool (which
this project does not define) or a CDK escape hatch to add `AWS_IAM` as an
additional auth provider. Neither was necessary for the PoC.

## Cross-domain contract

The app's Amplify backend writes four SSM parameters at deploy time, which
the scraper domain reads at synth time:

- `/reviewlensai/appsync/url` — AppSync GraphQL endpoint
- `/reviewlensai/appsync/apiId` — AppSync API ID
- `/reviewlensai/appsync/apiKey` — AppSync API key (for backend writers)
- `/reviewlensai/amplify/url` — the deployed hosting URL (used by the
  scraper's Validator Lambda Function URL for the CORS allow-list, and by
  Playwright E2E as the base URL)

The app receives the Validator's URL via the Amplify branch environment
variable `VITE_VALIDATOR_URL`, set by the scraper deploy workflow after the
scraper stack creates the Lambda Function URL.

## Local development

The SPA runs against MSW (handlers for the Validator URL) and a
`FakeAmplifyClient` (in-memory `Job` model that simulates the
`PENDING → RUNNING → SUCCEEDED` lifecycle on a timer). No AWS resources are
touched in local development. Staging is the only integration surface.

## Observability

CloudWatch log group for the Amplify build/deploy with 14-day retention.
