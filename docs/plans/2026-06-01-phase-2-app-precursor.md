# Phase 2 App-Precursor Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Upgrade the app's Amplify Gen 2 backend to the latest stable and add the three Phase-2 analytics fields to the `Job` schema, validated without breaking the live Phase-1 scrape flow.

**Architecture:** Two isolated changes in the **app domain only**, as separate commits for attributable CFN changesets: (1) dependency upgrade, (2) additive nullable schema fields. Pre-merge validation runs in an isolated Amplify **sandbox**; the scraper-compatibility proof runs **post-merge** on staging (a real scrape exercising the scraper's conditional `updateJob` mutations against the upgraded schema), with a revert-based rollback if it breaks.

**Tech Stack:** AWS Amplify Gen 2 (`@aws-amplify/backend`), `aws-amplify` client, AppSync, DynamoDB, GitHub Actions, AWS CLI, vitest.

**Spec:** `docs/specs/2026-06-01-phase-2-analytics-design.md` §2-A, §3, §16.

**Environment facts (project memory):** AWS CLI default profile, account `934939427723`, `us-east-1`. Amplify app id `d1zccfa4rl4qf6`. On Windows/Git-Bash, prefix any `aws ssm` call whose `--name` starts with `/` with `MSYS_NO_PATHCONV=1`. vitest runs with `pool: 'forks'`. **Each Bash invocation is a fresh shell — every command block below is self-contained (it recomputes the vars it needs); do not rely on vars set in a previous block.**

> **Corrections baked in from the plan-DA panel (spec §16):** (a) there is **no** SSM-republish step — `app/amplify/backend.ts:14-24` already creates `/reviewlensai/appsync/{url,apiKey}` as CFN-managed params; (b) `analyticsStatus` is `a.string()` (not `a.enum()`) so its `ModelStringInput` exposes `attributeExists` for the PR-3 idempotency guard.

```bash
VURL=$(MSYS_NO_PATHCONV=1 aws ssm get-parameter --name /reviewlensai/scraper/validatorUrl --query Parameter.Value --output text --region us-east-1) \
  && TABLE=$(aws dynamodb list-tables --region us-east-1 --query "TableNames[?starts_with(@,'Job-')]" --output text) \
  && JOB=$(curl -s -X POST "$VURL" -H 'content-type: application/json' -d '{"url":"https://store.steampowered.com/app/367520/Hollow_Knight/"}' | jq -r .jobId) \
  && echo "jobId=$JOB" \
  && for i in $(seq 1 36); do \
       ST=$(aws dynamodb get-item --table-name "$TABLE" --region us-east-1 --key "{\"id\":{\"S\":\"$JOB\"}}" --query "Item.status.S" --output text); \
       echo "status=$ST"; \
       [ "$ST" = "SUCCEEDED" ] && { echo "PASS: scraper conditional mutations OK on upgraded schema"; break; }; \
       [ "$ST" = "FAILED" ] && { echo "FAIL: job FAILED — inspect scraper logs"; break; }; \
       sleep 5; \
     done
```
Expected: `status` reaches `SUCCEEDED` and prints `PASS: ...`. This proves end-to-end that the upgraded AppSync schema still accepts the scraper's `UpdateJobInput` + `ModelJobConditionInput` conditional `updateJob` mutations (the surface most at risk from the upgrade).
